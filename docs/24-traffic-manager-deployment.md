# Traffic Manager Deployment Design

## 1. Overview

The traffic manager is a pod deployed in the cluster that provides the VPN server, envoy xDS control plane, and DNS forwarding. This document describes how `createOutboundPod` creates it programmatically and how `UpgradeDeploy` handles version upgrades.

## 2. Resource Set

`createOutboundPod` creates the following resources in order:

| Step | Resource | Name | Purpose |
|------|----------|------|---------|
| 1 | Namespace label | `ns: {namespace}` | For label-based selection |
| 2 | ServiceAccount | `kubevpn-traffic-manager` | Pod identity (`automountServiceAccountToken: true`) |
| 3 | Role | `kubevpn-traffic-manager` | RBAC: get/list/watch/create/update/patch/delete on configmaps + secrets (resourceName scoped) |
| 4 | RoleBinding | `kubevpn-traffic-manager` | Binds Role to ServiceAccount |
| 5 | Service | `kubevpn-traffic-manager` | ClusterIP: 10801/TCP, 9002/TCP, 53/UDP |
| 6 | ConfigMap | `kubevpn-traffic-manager` | Initial keys: ENVOY_CONFIG, DHCP, DHCP6, IPv4_POOLS |
| 7 | Secret | `kubevpn-traffic-manager` | TLS cert/key/server_name (Opaque type, env-var friendly keys) |
| 8 | Deployment | `kubevpn-traffic-manager` | 3 containers: vpn, control-plane, dns |

If any step fails, `cleanupTrafficManagerResources` deletes all resources (best-effort).

## 3. Deployment Spec (`genDeploySpec`)

### Containers

| Container | Command | Ports | Resources | Security |
|-----------|---------|-------|-----------|----------|
| `vpn` | `kubevpn server -l gtcp://:10801 -l gudp://:10802` | 10801/TCP | 500m-2000m CPU, 512Mi-2Gi mem | `privileged: false` |
| `control-plane` | `kubevpn control-plane` | 9002/TCP | 100m-200m CPU, 128Mi-256Mi mem | â€” |
| `dns` | `kubevpn dns` | 53/UDP | 100m-200m CPU, 128Mi-256Mi mem | â€” |

### Health Probes

vpn and control-plane containers have TCP socket probes:

```
Liveness:  TCP port, initialDelay=5s, period=15s, failureThreshold=3
Readiness: TCP port, initialDelay=3s, period=10s, failureThreshold=3
Startup:   TCP port, initialDelay=1s, period=2s,  failureThreshold=15
```

### Other Settings

- `sidecar.istio.io/inject: "false"` annotation (prevent Istio sidecar injection)
- `restartPolicy: Always`
- Selector: `app: kubevpn-traffic-manager`
- `envFrom: secretRef` on vpn container (TLS env vars)
- Optional `imagePullSecrets` if configured

## 4. TLS Secret

Generated by `GenTLSCert(ns)`:
- Self-signed cert with CN = `kubevpn-traffic-manager.{namespace}`
- Alternate IPs: `127.0.0.1`
- Alternate DNS: `localhost`
- Stored as `SecretTypeOpaque` (not `SecretTypeTls`) because key names use underscores (`tls_crt`, `tls_key`, `tls_server_name`) to be valid as shell env var names

## 5. Upgrade Flow (`UpgradeDeploy`)

When a newer client connects to an older traffic manager:

```
UpgradeDeploy:
  1. Get current Deployment + running pod
  2. Compare client version vs server image version (util.IsNewer)
  3. If upgrade needed:
     a. upgradeSecretSpec â€” ensure Secret has all 3 TLS fields
     b. upgradeServiceSpec â€” update Service ports to match genService
     c. kubectl set image â€” update all container images
     d. RolloutStatus â€” wait for rollout to complete
```

## 6. Existence Check

Before creating, `createOutboundPod` calls `util.DetectPodExists` to check if a traffic manager pod already exists. If it does, creation is skipped with "Use exist traffic manager" message.

## 7. Cleanup

`cleanupTrafficManagerResources` performs best-effort deletion of all resources with `GracePeriodSeconds: 0`:
- RoleBinding, Role, ServiceAccount, Service, ConfigMap, Secret
- CNI network pod (`config.CniNetName`)
- Job (Helm post-install hook)
- Deployment

## 8. Helm Chart vs Code

The Helm chart (`charts/kubevpn/`) creates the same resources but via templates. The chart also includes a `Job` (post-install hook running `kubevpn once`) for TLS cert generation and namespace labeling that `createOutboundPod` does inline.
